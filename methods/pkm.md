# Personal knowledge management structure

**Applicability:** any host with a persistent, file-based knowledge store it's expected to maintain over time.

- **Four-layer structure per topic area:** an entry-point doc (what's live, what matters right now) → `raw/` (unprocessed captures: transcripts, bookmarks, session logs) → a compressed/cross-linked layer (the actual reusable knowledge) → long-lived multi-file projects for anything that outgrows a single note.
- **Structured frontmatter on every knowledge file:** at minimum a title, a type, a date, a status, and tags — enough that an index can be generated from it mechanically rather than hand-maintained.
- **Regenerate the index after structural changes** (new files, renames, tag changes) — don't let it silently drift from reality. An index nobody trusts gets ignored, which defeats the point of having one.
- **A capture point for things that don't yet have an obvious home** — a single inbox, triaged later, rather than either forcing premature categorization or losing the thought.
- **Lessons/patterns extraction is a first-class habit, not an afterthought:** when something surprising or costly happens, write down what would have prevented it, in a place future work will actually consult before repeating the mistake.

# Lessons Learned

Experience knowledge from concrete work. Different from `learn`
(patterns/behavior rules in cross-session memory) and from a raw-capture
inbox: **lessons are condensed insights from mistakes actually made and
solutions actually found**, stored in the knowledge store so they're usable
across projects.

The storage layout below follows the consumer's PKM structure (see
`methods/pkm.md`): lessons live in the compressed/cross-linked knowledge
layer, in a `lessons/` subdirectory per topic area, plus one cross-cutting
lessons directory for insights that don't belong to a single area.

---

## Mode 1: EXTRACT

### When
- The user explicitly asks: "save lessons", "/lessons", "lesson learned"
- At the end of a session
- BEFORE the user runs "commit and push"

### What to extract
Scan the conversation for:
- **Mistake + fix sequences**: "X didn't work because Y, the solution was Z"
- **Discarded approaches**: "we tried this but it doesn't work because…"
- **Third-party gotchas**: undocumented API behavior, scope problems, silent failures
- **Process insights**: "should have checked X first", "next time do Y before Z"
- **The user's aha moments**: explicit statements like "ah, so that's why…", "interesting that…"

Do **not** extract: routine tasks, one-off bugs with no learning value,
anything already in memory or the knowledge store.

### Where to store

| Lesson type | Target |
|---|---|
| Clearly belongs to one topic area | that area's knowledge layer, `lessons/YYYY-MM-DD-<slug>.md` |
| Cross-cutting (AI workflow, deploy process, tooling) | the cross-cutting lessons directory |

Directories are created lazily (`mkdir -p` on first write).

### Frontmatter standard

```yaml
---
title: <one crisp sentence>
type: lesson
areas: [<one or more topic areas — cross-area matters>]
project: <slug>            # optional, if project-specific
tags: [<same taxonomy as the rest of the knowledge store>]
date: YYYY-MM-DD
status: active
source: <session context, ticket reference, whatever it was>
related: [<relative links to related notes>]   # optional
---
```

### Body structure (Rule + Why + How-to-apply)

```markdown
## What happened
<1-3 sentences: the concrete situation — the setting, what went wrong>

## Rule
<the generalized insight as a clear rule — written so it's still
understandable in 6 months>

## Why
<why the rule holds — context, system behavior, root cause — so it isn't
dismissed as dogma>

## How to apply
<concrete instruction for the future, ideally with a trigger condition or a
checklist item>
```

### Flow

1. **Scan the conversation** for lesson signals (above).
2. **Cluster** similar observations → one lesson per insight, not per remark.
3. **Ask the user briefly**: "I identified N lessons: [list]. Save?" — get
   confirmation for >3 lessons; for 1–2 save directly and just announce it.
4. **Write** one MD file per lesson in the right place.
5. **Update the knowledge index**: run the consumer's `indexer_command`
   (from `.methodik-capabilities.json`), if one is configured.
6. **Back-link into memory** (if the consumer maintains cross-session memory,
   `runtimes.<runtime>.memory_path`): if the lesson has a `project:`
   frontmatter field and a matching project memory file exists, append the
   lesson to that file's `## Lessons` section (create the section if
   missing). Format: `- [<title>](<relative path>) — <one line on what the
   lesson says>`.
7. **Confirm** in chat: "N lessons saved at [paths]. Index updated.
   Memory back-links set: [which files]."

### Slug convention
`YYYY-MM-DD-<short-and-crisp>.md` — e.g. `2026-04-25-css-specificity-vs-load-order.md`

## Mode 2: QUERY

### When
On ANY substantial question from the user that touches an area where lessons
could exist — deploys, migrations, browser automation, AI code generation,
third-party integrations, platform-specific quirks. Concretely: anything that
matches the tags already present in the lessons index.

### How

1. **Read the machine-readable knowledge index** (generated by the
   consumer's indexer).
2. **Filter** entries with `type: lesson` by:
   - tag match against the question (question mentions a platform → search
     that tag)
   - area/project match (question about project X → `project: x*`)
3. **Load** the top 1–3 relevant lesson files.
4. **Answer** with an explicit pointer: "Relevant lesson from [date]:
   [core point]" — then the actual answer.

### Fail silent
If no lessons match: no mention, answer normally. Lessons are a bonus, not a
mandatory block.

### Performance
The index scan is cheap (one file). Lesson files are small. When unsure,
prefer loading over skipping.

---

## Migration / backfill

If lessons already exist elsewhere (memory entries, old notes):
- memory entries with lesson character → migrate into the knowledge store's
  lessons layer
- then reduce the memory entry to a pointer: "see `<lessons path>/<slug>.md`"

## Example (good)

```markdown
---
title: Theme CSS beats component CSS via load order, not specificity
type: lesson
areas: [work]
project: storefront-rebuild
tags: [css, theme, frontend]
date: 2026-04-25
status: active
source: storefront rebuild session
---

## What happened
Component CSS was overridden by the shop theme's global styles. A broad
selector like `.nav p` beat the component's own class because of load order.

## Rule
Shop themes use aggressive global selectors everywhere. Component CSS inside
a theme needs **explicit values or high-specificity selectors**, or it gets
overridden.

## Why
The theme CSS is loaded by the bundler AFTER component CSS and therefore wins
"last write" ties. Specificity calculus alone isn't enough — load order beats
specificity when specificity is equal.

## How to apply
Before marking theme-component CSS as "done": **check computed styles in
DevTools**. AI is bad at judging specificity in context.
```

## Example (bad)

```markdown
## Bug fix
The CSS selector was wrong, changed it to the component class.
```
→ no learning value, no context, no generalization. Don't save.
